🔹 Domain & Functionality
Q1 DOMAIN
What is the core purpose of PixelForge?
Hint: Think about what the user experience is end-to-end.
Correct! PixelForge is a client-only SPA. Users enter a text prompt → select an LLM provider → the LLM returns structured pixel data → it's rendered as a 16×16 grid in the browser.
Q2 DOMAIN
How does the user's API key flow through the system?
Correct! The useApiKey hook manages the key via sessionStorage, so it lives for the tab session but is gone when the tab closes. The app never sends it anywhere except directly to the provider's API — no backend proxy.
Q3 DOMAIN RELATIONSHIP
The LLM response is parsed and validated before use. Which file validates the raw LLM output, and what type does it return?
Relationship chain: Every provider calls this parser. Its output feeds AvatarGrid.tsx for rendering.
Correct! parseResponse.ts exports parsePixelMapResponse(), which is called by every provider implementation to validate the LLM's JSON output. It returns a Result<PixelMapResponse> — either { ok: true, data } with validated palette/grid/label, or { ok: false, error } with a PARSE_ERROR code.
Q4 DOMAIN
What export options are available after generating an avatar?
Correct! Two export paths: PNG (via OffscreenCanvas → convertToBlob) and HTML (via buildAvatarHTML → self-contained document with inline styles, no external deps).

App.tsx composes several custom hooks. Which hooks manage (A) API key storage, (B) provider configuration, and (C) generation lifecycle?

Correct! Q6 RELATIONSHIP
Trace the generation call chain starting from user input. Which file dispatches to the correct LLM provider, and what pattern prevents concurrent calls?
Graph clue: Follow the imports edges: ControlPanel → fires callback → App.tsx calls hook → hook calls llmClient.generate() → dispatches by type.
Correct! The chain is: ControlPanel → App.tsx → usePixelGeneration.generate() → llmClient.generate(type, request). The llmClient looks up the provider in a lazy-initialised Record. Back in the hook, a useRef (generatingRef) prevents overlapping calls; it's cleared in a finally block.

Q8 RELATIONSHIP
All provider classes implement a common interface. Where is that interface defined, and what method does it require?
Graph clue: The graph has implements edges from each provider class to interface:LLMProvider. That interface node has defined_in: src/services/providers.ts.
Correct! The LLMProvider interface lives in src/services/providers.ts (just 5 lines). It defines a single method: generate(request: PixelGenRequest): Promise<Result<PixelMapResponse>>. All 5 providers — AnthropicProvider, OpenAIProvider, OpenAICompatibleProvider, DeepSeekProvider, GeminiProvider — implement this contract.
Q9 ARCHITECTURE RELATIONSHIP
How does DeepSeekProvider differ from the other providers that share the same API structure?
Graph clue: The graph lists DeepSeekProvider's summary — look for "truncation" and "thinking." Compare with OpenAIProvider's summary.
Correct! DeepSeekProvider uses the same /v1/chat/completions + Bearer auth as OpenAI, but adds two DeepSeek-specific features: it disables thinking mode and checks finish_reason === 'length' to detect truncated responses.
Q10 RELATIONSHIP

Q11 RELATIONSHIP
sanitize.ts is a small utility. Which two export functions import and use sanitize?
Graph clue: The graph's imports edges from utility files point to sanitize.ts. Follow them.
Correct! Both pngExport.ts and htmlExport.ts import sanitize to produce safe filenames. sanitize lowercases, removes non-alphanumeric chars (except hyphens), and collapses multiple hyphens — preventing issues with special characters in downloaded file names.

🏗️ Code Architecture
Q14 ARCHITECTURE RELATIONSHIP
The Result<T, E> type is a discriminated union that appears throughout the codebase. Which of the following best describes how it flows through the system?
Graph clue: The Result type node has edges from every function that returns it. Trace: provider → llmClient → hook → component.
Correct! Result<T, E> is defined in pixelmap.ts and used as the return type of the LLMProvider interface. Every provider returns Result<PixelMapResponse>, llmClient.generate() propagates it, and usePixelGeneration destructures it: ok: true → setPixelMap(data), ok: false → setError(error). The graph's type edges connect Result to every file that uses it.
Q15 ARCHITECTURE RELATIONSHIP
useProviderConfig.ts tracks "override" flags. Why is this necessary, and how do override flags relate to PROVIDER_PRESETS?
Graph clue: useProviderConfig.ts imports PROVIDER_PRESETS from pixelmap.ts. When a user switches providers, overrideUrl and overrideModel flags determine whether to overwrite with preset values.
Correct! When a user customises the API URL or model for one provider, then switches to another and switches back, the override flags (overrideUrl, overrideModel) ensure their custom values aren't silently replaced by the preset defaults. The hook auto-saves to localStorage and shows a "Settings saved locally" notice for 3 seconds. This is a UX detail that's easy to miss without understanding the relationship between provider presets and user config.
