# Real Session Evaluation

Evaluate opencode's actual responses to your real prompts using RAGAS metrics.

## The Problem

The existing pipeline uses predefined synthetic tasks (`generate_deepseek_tasks.py`) with
hardcoded contexts. These don't capture what the agent actually sees — the full project
state, conversation history, and tool outputs — so the evaluation may not reflect
real-world accuracy.

## The Approach

Instead of synthetic tasks, this workflow extracts conversation pairs directly from
opencode's SQLite database (`~/.local/share/opencode/opencode.db`). For each user
prompt → agent response pair, it captures:

- **user_input** — The actual prompt you typed
- **response** — The agent's final text response (aggregated across tool-call steps)
- **reasoning** — The agent's internal reasoning/thinking trace
- **retrieved_contexts** — The project state at session time (git branch, commit hash, dirty files), conversation history, and the actual source files the agent read via `read` tool calls
- **usage** — Token counts from the session

This gives you RAGAS faithfulness (and optionally precision/recall) scores on
real interactions rather than canned questions.

## Usage

### 1. List recent sessions

```bash
python3 .understand-anything/ragas/extract_opencode_sessions.py list
```

Shows session ID, title, agent type, token counts, and project directory.

Use `--limit 20` to show more, or `--tree` to see parent/child session hierarchy
(subagent tasks spawned from a parent conversation):

```bash
python3 .understand-anything/ragas/extract_opencode_sessions.py list --tree
```

### 2. Extract a session as RAGAS cases

```bash
python3 .understand-anything/ragas/extract_opencode_sessions.py extract <session_id> \
    --output my_cases.json
```

Produces a JSON file where each user→assistant turn is one evaluation case.

Extract multiple sessions (merged into one file):

```bash
python3 .understand-anything/ragas/extract_opencode_sessions.py extract <session_a> <session_b> \
    --output merged_cases.json
```

Include subagent sessions (child sessions spawned by `task` tool dispatches):

```bash
python3 .understand-anything/ragas/extract_opencode_sessions.py extract <session_id> \
    --with-children --output full_tree.json
```

### 3. Generate reference answers (optional, for all 4 metrics)

```bash
export DEEPSEEK_API_KEY="sk-..."
python3 .understand-anything/ragas/extract_opencode_sessions.py enrich \
    my_cases.json --output enriched_cases.json
```

This reads the extracted cases and calls DeepSeek to produce a `reference`
(ground truth) answer for each case using the `user_input` and captured context.
**Without this step, `context_precision` and `context_recall` will be `null`.**

The reference is generated by prompting DeepSeek with:

```
system: You are a senior software engineer. Given a question and project context,
        produce the definitive answer. Reference specific files, functions, and
        code patterns from the context.

user:   Question: <user_input>

        Context available to the agent:
        ────────────────────────────────────────────
        <retrieved_contexts[0]>
        <retrieved_contexts[1]>
        ...
        ────────────────────────────────────────────

        Write a comprehensive, accurate answer based on the context above.
```

Defaults: model `deepseek-v4-flash`, endpoint `https://api.deepseek.com`.
Override via `DEEPSEEK_MODEL` and `DEEPSEEK_BASE_URL` env vars.

### 4. Evaluate with RAGAS

Score the responses using the enriched (or plain extracted) file:

```bash
python3 .understand-anything/ragas/evaluate_deepseek_ragas.py \
    --input enriched_cases.json --output results.json
```

This computes:
- **Faithfulness** — Is the response grounded in the retrieved context?
- **Context Precision** — Are relevant context items ranked above irrelevant ones?
  *(requires `reference`)*
- **Context Recall** — Does the context contain all info needed to answer?
  *(requires `reference`)*
- **Answer Relevancy** — Is the response relevant to the question?
  *(requires `DEEPSEEK_EMBED_MODEL`)*

An HTML report is auto-generated alongside the JSON results at
`results.html` (same path as `--output` with `.html` extension). Pass
`--report report.html` to set a custom path, or `--report ''` to skip.
No separate `generate_compare_html.py` step is needed.

### All-in-one: extract → evaluate → HTML

```bash
export DEEPSEEK_API_KEY="sk-..."
python3 .understand-anything/ragas/extract_opencode_sessions.py eval <session_id> \
    --output-dir ./eval-results
```

This runs extract + evaluate + HTML in one command (faithfulness only — no reference).

To evaluate with references in one pass (enrich + evaluate + HTML):

```bash
python3 extract_opencode_sessions.py extract <session_id> --output /tmp/cases.json
python3 extract_opencode_sessions.py enrich /tmp/cases.json --output enriched.json
python3 evaluate_deepseek_ragas.py --input enriched.json --output results.json
# results.html is auto-generated
```

## Output structure

**Extracted (plain):**
```json
{
  "sessions": [
    { "id": "ses_...", "title": "...", "agent": "build", "cases": 5 }
  ],
  "total_usage": {
    "prompt_tokens": 5000,
    "completion_tokens": 3000,
    "total_tokens": 8000
  },
  "cases": [
    {
      "id": "ses_...__turn_1",
      "category": "complex",
      "context_type": "opencode_session",
      "user_input": "the prompt you typed...",
      "response": "the agent's answer...",
      "retrieved_contexts": [
        "=== Git State ===\ngit branch: main\n...",
        "=== File: src/services/llmClient.ts ===\n...",
        "=== File: src/types/pixelmap.ts ===\n..."
      ],
      "reasoning": "the agent's internal thinking...",
      "usage": {
        "prompt_tokens": 2500,
        "completion_tokens": 1500,
        "reasoning_tokens": 200,
        "total_tokens": 4000
      }
    }
  ]
}
```

**After enrich** — each case gains a `reference` field:
```json
{
  "cases": [
    {
      "id": "ses_...__turn_1",
      "user_input": "...",
      "reference": "Add 'groq' to LLMProviderType in pixelmap.ts...",
      "retrieved_contexts": [...],
      ...
    }
  ]
}
```

**Evaluated result** — each case gains a `scores` map:
```json
{
  "results": [
    {
      "id": "ses_...__turn_1",
      "scores": {
        "faithfulness": 0.85,
        "context_precision": 0.72,
        "context_recall": 0.64,
        "answer_relevancy": 0.91
      }
    }
  ]
}
```

## Interpretation

| Score | Range | Meaning |
|---|---|---|
| Faithfulness | 0–1 | How much of the response is directly supported by the retrieved context |
| Context Precision | 0–1 | Whether the most relevant context items appear first (`null` without `reference`) |
| Context Recall | 0–1 | Whether all necessary information is present in the context (`null` without `reference`) |
| Answer Relevancy | 0–1 | Whether the response addresses the question directly |

A low faithfulness with high context recall suggests the agent ignored available
context. Low context recall suggests the agent didn't have the right files
available. Low context precision suggests the context contained noise.

## Limitations

- **No ground truth** — Real prompts don't have a `reference` answer by default. Use
  the `enrich` subcommand to generate references via DeepSeek, enabling all 4 metrics.
  Generated references are approximate — they reflect what an LLM considers correct
  based on the retrieved context, not a human-written ground truth.
- **Context is approximate** — The script captures git state, conversation
  history, and files the agent read via `read` tool calls. `grep`/`bash`/`edit`
  tool outputs are not yet included. File contents are truncated to 30k chars.
- **Only text responses** — The agent's tool calls are not included as response
  text; only the final textual output is used.
- **DB must be accessible** — opencode's database may be locked if opencode is
  actively writing. Wait a moment after closing opencode before running extraction.
- **No embedding model configured** — Answer Relevancy requires `DEEPSEEK_EMBED_MODEL`
  to be set. Without it, this metric is skipped.

## Comparison to synthetic task pipeline

| Aspect | Synthetic (`generate_deepseek_tasks.py`) | Real session (this workflow) |
|---|---|---|
| Prompts | Predefined coding tasks | Your actual prompts |
| Context | Hand-picked source files | Git state + conversation history + files agent read |
| Reference | Yes (human-written) | LLM-generated via `enrich` |
| Metrics | All 4 RAGAS metrics | All 4 (with `enrich`) or Faithfulness + Answer Relevancy |
| Use case | Benchmarking KG impact | Measuring real agent accuracy |
